### CS250 — Arrays & Hashing: Two Sum

**Concept-first lesson (hash maps turn O(n²) into O(n))**

The brute-force approach to "find two numbers that sum to target" checks every pair: O(n²). The insight that makes this O(n) is the **complement trick**: for each number `x`, you're looking for `target - x`. If you store every number you've seen in a hash map (value → index), you can check "have I already seen the complement?" in O(1) per element.

Mental model: walk the array once. At each step, ask the hash map "did I already pass the number I need?" If yes → return both indices. If no → put current number + its index into the map so a future element can find it.

This is the **space-for-time trade** in its simplest form — you spend O(n) extra memory to buy O(n) time instead of O(n²).

**Task:** Solve [Two Sum](https://leetcode.com/problems/two-sum/) on LeetCode. Target: O(n) with a hash map, not the O(n²) brute force. After AC, write a **3-line pattern note**: what's the pattern, why it applies, and what's the invariant (the complement relationship). Time budget: ~30 min.

### CS301 — System Design: 6-Step Delivery Framework

**Concept-first lesson (structure beats brilliance under time pressure)**

System design interviews aren't about the perfect architecture — they're about **showing your work in an evaluable way**. The Hello-Interview 6-step delivery framework gives you a rails:

1. **Requirements** (2–3 min): clarify functional + non-functional. "What are we building? How many users? Read-heavy or write-heavy?"
2. **Entities / data model** (2 min): what are the nouns? User, Tweet, Follow — sketch the core tables.
3. **API design** (3 min): REST endpoints the client calls. `POST /tweet`, `GET /feed`.
4. **High-level design** (5 min): boxes and arrows — load balancer → app servers → DB → cache. Draw it.
5. **Deep dive** (10–15 min): pick the bottleneck (usually the DB) and scale it — sharding, caching, replication.
6. **Wrap-up** (2 min): name one trade-off you made and one thing you'd improve with more time.

The framework's power: it prevents rambling, guarantees you hit every signal the interviewer is scoring, and keeps you inside the time box. **Every minute past step 2 without a diagram is a minute the interviewer can't evaluate you.**

**Task:** Take the prompt "Design a URL shortener" and apply the framework. Write a design doc (`Courses/CS301/designs/url-shortener-framework.md`) that explicitly works through each of the 6 steps, with an HLD diagram (ASCII or boxes-and-arrows text). Include a BOTEC: how many URLs stored per month, QPS, storage estimate. Keep each step to its time-boxed length — this is about practicing the rhythm, not writing a novel.

**Proof gate:** Design doc with all 6 steps + BOTEC.
