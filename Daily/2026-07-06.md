**Difficulty:** med | **Scaffold:** independent | **Proof:** a.ah.solve (LeetCode AC + pattern note)

#### Concept Lesson: The Hash Map as an O(1) Lookup Table

The core insight: a hash map turns "find X in a collection" from O(n) scanning into O(1) amortized lookup. In array problems, brute-force solutions almost always involve a nested loop — for each element, scan the rest. A hash map eliminates the inner loop by storing values you've already seen, enabling a single pass.

**Mental model:** You're walking through the array. At each step, instead of asking "does the complement exist somewhere ahead?", you ask "have I already seen the complement?" The hash map is your memory of everything you've passed.

**Key invariant for today:** `target - nums[i]` was either seen before (found it) or it wasn't (store `nums[i]` and keep going). This is the **complement lookup pattern** — the most common hash map interview pattern.

**Space-for-time trade:** You spend O(n) memory on the hash map to buy O(n) time instead of O(n²). In interview problems, this is almost always the right trade — memory is cheaper than CPU cycles.

**Python hash map operations:**
```python
seen = {}                    # dict is a hash map
seen[key] = value            # O(1) insert/update
if key in seen: ...          # O(1) membership
val = seen.get(key, default) # O(1) safe lookup
seen = set()                 # for presence-only tracking
```

**When to reach for a hash map:**
- "Find two numbers that sum to X" → complement lookup
- "Check if element exists" → O(1) membership
- "Count frequencies" → `collections.Counter` (hash map under the hood)
- "Group by a key" → `defaultdict(list)`
- "First occurrence / earliest index" → store index as value

**Pitfalls:** hash collisions degrade to O(n) worst case (rare with Python's implementation). Don't use a hash map when the problem asks for O(1) space — that means two-pointers or in-place.

#### Concept Lesson: The Hello-Interview Delivery Framework

System design interviews are a **performance**, not a lecture. The 6-step framework (from Hello-Interview) gives you a reliable structure that signals senior-level thinking in 45 minutes:

```
 1. Requirements    (5 min)  — clarify functional + non-functional, scope down
 2. Entities        (3 min)  — identify core data objects and their relationships
 3. API Design      (5 min)  — sketch the REST/gRPC surface (just the key endpoints)
 4. High-Level Design (10 min) — boxes-and-arrows diagram of the system
 5. Deep Dive       (15 min) — pick ONE component and go deep (the bottleneck)
 6. Wrap-up         (5 min)  — failure modes, monitoring, what breaks at 10x
```

**Why this order:** Requirements first prevents the #1 interview-killer — building the wrong thing. Entities force you to name your data before you design around it. API grounds the design in something concrete. HLD gives the interviewer a map. The deep dive is where you prove depth. Wrap-up shows operational maturity.

**The secret:** You don't need to design everything. You need to **scope aggressively** in step 1, then go deep on one component in step 5. The interviewer is evaluating your judgment, not your diagram.

**Key signal:** Always start with numbers — "How many users? How many requests per second? How much data?" BOTEC (Back of the Envelope Calculation) belongs in step 1, not as an afterthought. Numbers drive every decision downstream.

*Source: Hello-Interview system design guide — the 6-step delivery framework.*
