# AG201 — AI Agent Engineering

> Design, evaluate, and deploy production-grade AI agent systems — from a single ReAct loop to multi-agent orchestrations — with the architectural judgment of a staff-level engineer who steers coding agents, not types every line.

Concept-first, eval-forward: a 22-week course that builds architectural judgment for the AI coding agents era. The learner already has a coding agent — this course teaches what to build, why, and how to evaluate it. Anchored in Chip Huyen's AI Engineering, Anthropic's engineering essays, and the Berkeley LLM Agents MOOC. Every unit teaches the mental model first, then asks the learner to critique, design, or evaluate — leaving implementation to their agent. Threads eval literacy from week 7 onward, with a dedicated observability deep-dive before the capstone. Graduates can defend an architecture decision record with quantitative reasoning about cost, latency, reliability, and when NOT to add complexity.

## What the best in this field can do
A staff-level agent engineer can: (1) decompose an ambiguous problem into the right control-flow shape (prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer); (2) design evaluation and observability infrastructure that makes iteration possible; (3) reason quantitatively about cost, latency, and reliability trade-offs; (4) design agent harnesses that survive model upgrades; (5) know when NOT to add agentic complexity; (6) read and critique a frontier agent architecture from primary sources; (7) define the problem and eval spec, then delegate implementation to a coding agent while reviewing output rigorously.

**How the best practice:**
- Define the eval spec before building anything — held-out test set, LLM-as-judge rubric, error categories. Then delegate the build, review the output, and iterate based on error analysis.
- Track total runtime per tool call, total tool calls, total token consumption, tool error rates, KV-cache hit rate, and trajectory quality — not just top-line accuracy.
- End-state evaluation for stateful agents, not turn-by-turn. LLM-as-judge for scalable scoring plus mandatory human review to catch edge cases.
- Keep architecture simple. Make reasoning visible. Ensure reliable, well-scoped tool interactions. Use checkpointing, retry logic, and incremental deployments.
- Design harnesses expecting to re-architect repeatedly as models improve. The willingness to strip away scaffolding when models internalize it is the staff-level muscle.
- Read one frontier paper per week, one production post-mortem per week. The field moves fast — architectural judgment comes from seeing what breaks at scale, not from tutorials.

**Signature work that earns a seat with the best:** An architecture decision record (ADR) for a production-grade agent system — designed, not just built. The ADR covers: problem decomposition into the right control-flow shape, tool selection and MCP integration strategy, context-engineering approach (with quantitative KV-cache reasoning), eval harness design (held-out sets, LLM-as-judge, error categories), observability plan (Langfuse tracing, key metrics, alert thresholds), and cost/latency/reliability tradeoffs defended quantitatively. The learner delegates implementation to their coding agent but reviews every output against the ADR. This is what earns a seat with the best — not a certificate, but demonstrable architectural judgment.

## How to keep evolving
The central live debate (2025–2026): single-agent (Cognition) vs. multi-agent (Anthropic). Anthropic's multi-agent system outperforms single-agent by 90.2% on internal evals but uses 15× more tokens. Code-execution agents (CodeAct) increasingly outperform JSON tool-calling. MCP has become the de-facto standard for agent-tool integration, adopted across the industry. SWE-bench Verified is approaching saturation (Claude Opus 4.5 at 79.2%) — the field is moving to harder benchmarks (SWE-bench Pro, Terminal-Bench). Agentic RL (AgentGym-RL, RAGEN) is the next frontier — training agents, not just prompting them. Shunyu Yao's "The Second Half" frames the era: evaluation over training, problem definition over problem solving.

## How this course is taught
*Concept-first, critique-second, evaluate-third. Every new idea starts with the mental model (diagram + primary source), then asks the learner to critique existing implementations or design their own — using their coding agent for the build work. The learner's job is architectural reasoning, eval design, and output review; the agent handles implementation. Faded scaffolding: heavily guided on foundations, increasingly independent as the learner builds judgment. Wrong answers get targeted micro-lessons on the exact misconception, then re-testing. Every unit connects back to a real production system (Anthropic, Manus, Fin).*

**Misconceptions the professor preempts:**
- Agents are just LLMs with tools wired in — the real complexity is context management, error recovery, and eval, not the wiring.
- Multi-agent is always better — single-agent beats multi-agent for tightly coupled work, and multi-agent's cost (15× tokens) must be justified by the task structure.
- You need LangChain/CrewAI to build agents — the best engineers understand the architecture first; frameworks earn their place later.
- Eval can wait until the end — eval is the engine of improvement and must be built alongside the agent from day one.
- More context = better agent — KV-cache hit rate is the single most important metric for production agent cost and latency; compaction beats expansion.
- In the coding-agents era, you don't need to understand code — you must read, review, and reason about code fluently; you just don't need to type every line.

## Enduring understandings
- Evaluation is the engine of improvement — not a final step. The #1 differentiator between competent and excellent agent engineers is disciplined eval and error analysis. Per Yao: the second half of AI shifts focus from solving problems to defining them.
- Context engineering — KV-cache economics, compaction, externalized memory, preserving failures — is where reliability is won or lost, not prompt polishing.
- Architecture follows task structure. Multi-agent wins for parallelizable, independent subtasks (but costs 15× tokens). Single-agent or deterministic pipelines win for tightly coupled work. Know when NOT to add agentic complexity.
- The Bitter Lesson applies at the application layer: scaffolding that models can internalize will be obsolete. Build harnesses that survive model upgrades. The willingness to strip scaffolding is the staff-level muscle.
- Compound errors in agentic systems mean minor issues can derail entire runs (95% per-step → ~60% over 10 steps). Observability, checkpointing, and graceful error recovery are not optional — they are the system.
- In the coding-agents era, the engineer's job shifts from writing code to defining problems, designing architectures, writing eval specs, and reviewing agent output. Code literacy is required; typing speed is not.
